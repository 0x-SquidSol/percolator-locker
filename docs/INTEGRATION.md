# Locker Integration

Authoritative reference for sibling repos (sdk, launch, indexer, matcher) that need to read, decode, or react to state owned by `percolator-locker`. This file documents only what this repo owns — account bytes, event payloads, PDA derivations, and the contracts those surfaces are committed to preserve.

## 1. Context

The locker program is the single on-chain source of truth for fee-discount entitlement. Users lock $PERCOLATOR into a `LockVault`; the program stamps an earned tier onto a per-user `LockPosition` and emits `Locked`, `Refreshed`, and `Unlocked` events. Downstream consumers read the position's `tier` and `discount_end` to gate discount application. Vault-level knobs (`lock_duration`, tier thresholds) are admin-mutable via `update_config` but position-level snapshots are immutable for the life of the position by design — see section 7.

Sibling stacks plug in at distinct layers. The SDK wraps the IDL and exposes typed client helpers. The launch site authors transactions against those helpers. The indexer subscribes to program logs, decodes events, and surfaces them. The matcher reads `LockPosition` accounts at fill time to apply the fee discount. All four must pin to the canonical `LockVault` pubkey (section 2) and treat every other `LockVault` under this program ID as noise.

## 2. Authoritative References

- **IDL:** `target/idl/percolator_locker.json`, generated by `anchor build`. This is the canonical machine-readable shape of every instruction, account, event, and error code. Pin the SDK build to a specific Anchor version (0.31.1) and regenerate on every `lib.rs`/`state.rs`/`error.rs` change.
- **TypeScript types:** `target/types/percolator_locker.ts`, generated alongside the IDL.
- **Program ID:** `91JU1rmiLAPNcmC9Kew8cCXTRGFW1Pe67ZreijUia5S8` (current `declare_id!`). This value is **pre-mainnet and subject to change** until mainnet deployment lands — do not ship it as a hardcoded constant without a feature flag or config override.
- **Canonical vault rule:** `initialize_vault` is permissionless. Multiple `LockVault` accounts coexist under the program ID and emit structurally identical events. Only one vault is canonical. Consumers **must filter by the canonical `LockVault` pubkey**, not by program ID alone. Full rationale: `README.md` (For Integrators) and the `LockVault` doc comment in `programs/percolator-locker/src/state.rs`. The canonical pubkey will be published in the README at mainnet.
- **Tier → discount mapping:** `Tier::discount_bps` in `programs/percolator-locker/src/state.rs` is the single canonical source. Values are protocol policy — not admin-adjustable — and match the README's Fee Discount Tiers table: `None=0`, `Bronze=1000`, `Silver=2000`, `Gold=3000` bps. Consumers must call this (or mirror its exact output) rather than re-deriving from tier names.

## 3. Byte Layouts

Both account types are Anchor-serialized with an 8-byte discriminator prefix. Offsets are stable under the change protocol in section 6. All multi-byte integers are **little-endian** (Borsh). Pubkeys are 32 raw bytes.

### 3.1 `LockVault` — 162 bytes total

| Offset | Size | Field                  | Type     | Notes |
|-------:|-----:|------------------------|----------|-------|
|      0 |    8 | discriminator          | `[u8;8]` | Anchor account discriminator |
|      8 |   32 | `admin`                | `Pubkey` | Vault admin, also in `LockVault` PDA seeds |
|     40 |   32 | `token_mint`           | `Pubkey` | Classic SPL Token mint (not Token-2022) |
|     72 |   32 | `vault_token_account`  | `Pubkey` | PDA-owned token custody account |
|    104 |    8 | `total_locked`         | `u64`    | Aggregate locked base units |
|    112 |    8 | `lock_duration`        | `u64`    | Seconds; cycle length for FUTURE locks only |
|    120 |    8 | `tier_bronze`          | `u64`    | Bronze threshold (base units) |
|    128 |    8 | `tier_silver`          | `u64`    | Silver threshold (base units) |
|    136 |    8 | `tier_gold`            | `u64`    | Gold threshold (base units) |
|    144 |    8 | `total_lockers`        | `u64`    | Count of active positions |
|    152 |    1 | `token_decimals`       | `u8`     | Cached from mint at init |
|    153 |    1 | `bump`                 | `u8`     | PDA bump |
|    154 |    8 | `last_config_update`   | `i64`    | Unix ts of last `update_config`; 0 if never |

### 3.2 `LockPosition` — 115 bytes total

| Offset | Size | Field            | Type     | Notes |
|-------:|-----:|------------------|----------|-------|
|      0 |    8 | discriminator    | `[u8;8]` | Anchor account discriminator |
|      8 |   32 | `owner`          | `Pubkey` | Position owner |
|     40 |   32 | `vault`          | `Pubkey` | Parent `LockVault` pubkey |
|     72 |    8 | `amount`         | `u64`    | Current locked base units; 0 after unlock |
|     80 |    8 | `lock_start`     | `i64`    | Original lock unix ts; immutable |
|     88 |    8 | `lock_end`       | `i64`    | Unlockable-at unix ts; advanced by `refresh_lock` |
|     96 |    8 | `discount_end`   | `i64`    | Earned-discount expiry unix ts |
|    104 |    1 | `tier`           | `u8`     | `Tier` enum Borsh variant index (0/1/2/3) |
|    105 |    1 | `is_active`      | `bool`   | 0x00 or 0x01; false after unlock |
|    106 |    1 | `bump`           | `u8`     | PDA bump |
|    107 |    8 | `cycle_duration` | `u64`    | Snapshot of `vault.lock_duration` at lock time; immutable |

`tier` encoding: `0 = None`, `1 = Bronze`, `2 = Silver`, `3 = Gold`. Zero-init decodes to `None` by design.

## 4. Event Payloads

Events are Anchor `emit!` payloads, prefixed with an 8-byte event discriminator when decoded from program logs. Field order matches declaration order. `tier` is a 1-byte Borsh variant index, same encoding as `LockPosition.tier`.

Every event carries a `vault: Pubkey` field. **Consumers must filter on the canonical vault pubkey** — filtering by program ID alone ingests events from uncontrolled vaults (section 2).

### 4.1 `Locked` — 8 fields

Emitted on every successful `lock`.

| Field            | Type     |
|------------------|----------|
| `user`           | `Pubkey` |
| `vault`          | `Pubkey` |
| `amount`         | `u64`    |
| `tier`           | `Tier`   |
| `lock_start`     | `i64`    |
| `lock_end`       | `i64`    |
| `discount_end`   | `i64`    |
| `cycle_duration` | `u64`    |

### 4.2 `Refreshed` — 8 fields

Emitted on every successful `refresh_lock`. Structurally identical to `Locked` — indexers can treat the two symmetrically as "cycle commitments." `lock_start` carries the ORIGINAL lock timestamp (refresh does not modify it); `lock_end` and `discount_end` are the NEW post-refresh values.

| Field            | Type     |
|------------------|----------|
| `user`           | `Pubkey` |
| `vault`          | `Pubkey` |
| `amount`         | `u64`    |
| `tier`           | `Tier`   |
| `lock_start`     | `i64`    |
| `lock_end`       | `i64`    |
| `discount_end`   | `i64`    |
| `cycle_duration` | `u64`    |

### 4.3 `Unlocked` — 7 fields

Emitted on every successful `unlock`. Note: **no `cycle_duration` field** — unlock retires the position and nothing downstream needs to re-derive cycle math from an unlock event. Indexers that share a decoder struct between `Locked`/`Refreshed` and `Unlocked` must account for this one-field shape difference.

| Field          | Type     |
|----------------|----------|
| `user`         | `Pubkey` |
| `vault`        | `Pubkey` |
| `amount`       | `u64`    |
| `tier`         | `Tier`   |
| `lock_start`   | `i64`    |
| `lock_end`     | `i64`    |
| `discount_end` | `i64`    |

### 4.4 `ConfigUpdated` — 7 fields

Emitted on every successful `update_config`. Carries the FULL resulting config (not just changed fields) so indexers can reconstruct vault state without a follow-up account read.

| Field           | Type     |
|-----------------|----------|
| `vault`         | `Pubkey` |
| `admin`         | `Pubkey` |
| `lock_duration` | `u64`    |
| `tier_bronze`   | `u64`    |
| `tier_silver`   | `u64`    |
| `tier_gold`     | `u64`    |
| `timestamp`     | `i64`    |

## 5. PDA Derivations

Seed byte strings are exported as `#[constant]` in `programs/percolator-locker/src/constants.rs` and appear in the IDL, so clients should read them from there rather than hardcoding.

- **`LockVault`**
  - Seeds: `[b"lock_vault", admin]`
  - `admin` is the 32-byte signer pubkey that called `initialize_vault`.
  - Stored bump: `LockVault.bump` (offset 153).
- **`LockPosition`**
  - Seeds: `[b"lock_position", vault, user]`
  - `vault` is the 32-byte `LockVault` pubkey; `user` is the 32-byte owner pubkey.
  - Stored bump: `LockPosition.bump` (offset 106).
- **`vault_token_account`**
  - Not a PDA of this program — a freshly-created SPL Token account whose authority is the `LockVault` PDA. Pubkey is stored at `LockVault` offset 72 and is write-once at init.

## 6. Change Protocol

These surfaces are committed to append-only evolution. Consumers that honor the rule can rely on existing offsets, event shapes, and error codes across program upgrades.

- **`Tier` enum:** Variants are append-only. Never insert mid-list, never renumber. Existing `LockPosition.tier` bytes decode by Borsh variant index — inserting a variant shifts every later index and misdecodes persisted state. New tiers go at the end of the enum.
- **Account fields (`LockVault`, `LockPosition`):** New fields are appended at the END of the struct. Never insert mid-struct, never reorder, never resize an existing field. `last_config_update` (on `LockVault`) and `cycle_duration` (on `LockPosition`) were added under this rule and sit at the tail of their respective structs — they do not shift any preceding offset. Consumers that decode by byte offset can rely on offsets 0..152 of `LockVault` and 0..107 of `LockPosition` remaining stable.
- **Error variants (`LockerError`):** Append-only. Anchor auto-numbers variants starting at 6000 by declaration order, so inserting mid-enum shifts every subsequent code and breaks clients pinned to numeric codes. New variants go at the end even if they belong semantically with an earlier section.
- **Event fields:** Append-only within each event. Never remove a field, never reorder, never change a field's type. The field-count asymmetry between `Unlocked` (7) and `Locked`/`Refreshed` (8) is intentional and will not be "normalized."

## 7. Matcher Snapshot Rule

**Read `LockPosition.tier` at offset 104. Do not re-classify from `amount` against the vault's current tier thresholds.**

Vault thresholds (`tier_bronze`, `tier_silver`, `tier_gold`) are admin-mutable via `update_config`. A matcher that re-classifies a position's `amount` against the live vault would retroactively change a user's discount when the admin tunes thresholds — exactly the outcome the program is designed to prevent. `LockPosition.tier` is stamped once at lock time from the vault state THEN in effect, and no handler (including `unlock` and `refresh_lock`) ever rewrites it. The same snapshot guarantee applies to `cycle_duration` (offset 107): `refresh_lock` advances by the position's snapshotted cycle, never by the vault's current `lock_duration`.

For the matcher's fee path:
1. Derive the `LockPosition` PDA for `(canonical_vault, user)` — seeds in section 5.
2. If the account does not exist, the user has never locked in this vault → no discount.
3. Read `is_active` (offset 105), `tier` (offset 104), and `discount_end` (offset 96).
4. The user is entitled to the discount iff `now < discount_end`. `is_active` is NOT required — unlock zeroes `amount` and flips `is_active` to false but preserves `tier` and `discount_end` precisely so the matcher can keep honoring the earned-discount window after withdrawal.
5. Translate `tier` to bps via the canonical `Tier::discount_bps` mapping (section 2).
